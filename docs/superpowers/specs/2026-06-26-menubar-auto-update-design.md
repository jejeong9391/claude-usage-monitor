# Claude Usage Monitor — 인앱 업데이트 & 재시작 설계

- 작성일: 2026-06-26
- 대상: `claude-usage-monitor` (로컬 전용 Swift 메뉴바 앱)
- 상태: 설계 승인됨

## 목표

로컬(개발자 본인 Mac)에서만 쓰는 메뉴바 앱에 두 가지를 정리·추가한다.

1. **직접 종료/시작** — 현재 구조를 유지한다. 추가 UI 없음.
2. **인앱 업데이트** — 팝오버에 "업데이트" 버튼을 추가해, 클릭 한 번으로 `git pull → 재빌드 → 재설치 → 재시작`이 자동으로 일어나게 한다.

배포·코드서명(Developer ID)·DMG는 범위 밖이다. ad-hoc 서명만 유지한다.

## 비목표 (YAGNI)

- `/Applications`(시스템 폴더) 설치 — `sudo`가 필요해 무인 업데이트와 충돌하므로 하지 않는다.
- 자동 업데이트 폴링/주기 확인 — 버튼 기반 수동 트리거만.
- 소스 파일 변경 감지(watcher) 기반 자동 재빌드 — 채택하지 않음.
- 외부 배포/공유 기능.

## 1. 설치 위치 — `~/Applications` 유지

`/Applications`는 쓰기에 관리자 권한이 필요해, 인앱 업데이트 버튼이 매번 비밀번호 프롬프트를 띄워야 한다. `~/Applications`는 권한 문제가 없고 Launchpad·Spotlight 노출도 동일하다(이미 확인됨). 따라서 설치 위치는 `~/Applications/ClaudeUsageMonitor.app`로 유지한다. `build.sh`의 기존 설치 로직을 그대로 사용한다.

## 2. 종료 / 시작 — 변경 없음

- **시작**: Launchpad / Spotlight에서 "Claude Usage Monitor" 검색, 또는 Finder에서 더블클릭.
- **종료**: 메뉴바 `🔥` 아이콘 클릭 → 팝오버 하단의 기존 종료 버튼(`onQuit` → `NSApp.terminate`).
- **로그인 자동시작**: 현재 로그인 항목에 등록되어 있음 — 그대로 유지(이 작업 범위 밖).

이 영역은 코드 변경이 없다.

## 3. 인앱 업데이트 (핵심)

### 3.1 UI

- 팝오버 하단, 기존 종료 버튼 옆에 "업데이트" 버튼을 추가한다.
- 상태 3가지를 버튼/라벨로 표현한다:
  - 대기(기본): "업데이트"
  - 진행 중: "업데이트 중…" (버튼 비활성)
  - 결과: 성공 시 재시작되므로 표시 불필요, 실패 시 에러 메시지 노출.

### 3.2 동작 순서

업데이트 버튼 클릭 시, 앱은 업데이트 스크립트를 백그라운드 프로세스로 실행한다. 스크립트는 다음을 수행한다.

1. 소스 루트로 이동.
2. `git pull --ff-only` — 원격에 새 커밋이 있으면 fast-forward로 당긴다. 충돌·인증 실패·로컬 미커밋 변경 등으로 실패하면 **무시하고 계속 진행**한다(로컬 워킹트리 기준 빌드를 막지 않기 위함).
3. `./build.sh` — **항상 현재 워킹트리 소스 기준으로** `swiftc -O` 재빌드 + ad-hoc 서명 + `~/Applications` 재설치. 즉 로컬에서 직접 수정한 코드도 반영된다.
4. 빌드 성공 → 새 버전으로 재시작(relaunch).
5. 빌드 실패 → 재시작하지 않고 에러를 앱에 전달, **기존(현재 실행 중) 앱을 그대로 유지**.

### 3.3 소스 경로 인지

실행 중인 `.app` 번들은 자기 소스 디렉토리 경로를 알아야 한다.

- `build.sh`가 빌드 시점에 소스 루트(`$ROOT`) 절대경로를 `Info.plist`의 커스텀 키 `SourceRoot`에 기록한다.
- 앱은 `Bundle.main.object(forInfoDictionaryKey: "SourceRoot")`로 그 경로를 읽어 업데이트 스크립트를 실행한다.
- `SourceRoot`가 없거나 디렉토리가 존재하지 않으면 업데이트 버튼은 비활성/에러 처리한다.

### 3.4 재시작(relaunch)

실행 중인 바이너리를 덮어써도 macOS는 inode 유지로 현재 프로세스가 죽지 않는다. 새 바이너리로 교체하려면 명시적 재시작이 필요하다.

- 빌드 성공 후, 분리된(detached) 프로세스를 띄워 짧은 지연 뒤 `open ~/Applications/ClaudeUsageMonitor.app`을 실행하게 하고, 현재 인스턴스는 `NSApp.terminate(nil)`로 종료한다.
- `LSUIElement` 앱은 동일 번들이 실행 중이면 `open`이 새 인스턴스를 띄우지 않을 수 있으므로, 현재 인스턴스 종료가 새 인스턴스 기동보다 먼저 일어나도록 지연/순서를 둔다.

### 3.5 에러 처리

- 스크립트는 단계별 종료 코드와 메시지를 표준출력/표준에러로 남긴다.
- 앱은 빌드 단계 실패를 구분해, 사용자에게 "업데이트 실패"와 원인 요약(예: 컴파일 오류)을 보여준다.
- 실패 시 현재 앱은 계속 동작한다(가용성 우선).

## 4. 컴포넌트 / 변경 파일

- `src/Views.swift` — 팝오버에 업데이트 버튼 + 진행/에러 상태 UI 추가.
- `src/main.swift` 또는 신규 `src/UpdateService.swift` — 업데이트 스크립트 실행, 결과 파싱, relaunch 로직. (업데이트 책임이 충분히 크면 별도 파일로 분리)
- `build.sh` — (a) `Info.plist`에 `SourceRoot` 주입, (b) 업데이트 스크립트를 앱 번들 `Contents/Resources/`에 포함하거나 소스 루트의 스크립트를 그대로 사용.
- 업데이트 스크립트(신규, 예: `update.sh`) — `git pull --ff-only`(실패 허용) + `build.sh` 호출. relaunch는 앱이 담당할지 스크립트가 담당할지는 구현 계획에서 확정.

### 인터페이스 경계

- **앱 ↔ 스크립트**: 앱은 스크립트 경로와 작업 디렉토리만 알면 된다. 결과는 종료 코드 + 마지막 출력으로 전달. 빌드 방법(swiftc 옵션 등)은 전적으로 `build.sh`에 캡슐화.
- **build.sh ↔ Info.plist**: `SourceRoot` 한 개 키로만 결합. 앱은 이 키 외에 빌드 내부를 알지 못한다.

## 5. 테스트 (로컬 수동)

1. **정상 경로**: 소스에서 메뉴바 타이틀 문구 등 한 줄 변경 → 업데이트 버튼 클릭 → 앱이 새 버전으로 재시작되고 변경이 반영되는지 확인.
2. **빌드 실패 경로**: 소스에 일부러 컴파일 오류 삽입 → 업데이트 버튼 → "업데이트 실패" 표시 + 기존 앱 유지(재시작 안 함) 확인. 이후 오류 수정 후 다시 업데이트하면 정상 반영.
3. **pull 스킵 경로**: 로컬에 미커밋 변경이 있는 상태에서 업데이트 → `git pull` 단계가 실패해도 현재 소스 기준으로 빌드·재시작되는지 확인.

## 6. 미해결/구현 단계에서 결정할 사항

- relaunch 책임 위치(앱 vs 스크립트)와 정확한 지연 시간.
- 업데이트 스크립트를 번들에 복사할지, 소스 루트의 것을 직접 실행할지.
- 에러 메시지 표면 방식(팝오버 내 라벨 vs `NSAlert`).
